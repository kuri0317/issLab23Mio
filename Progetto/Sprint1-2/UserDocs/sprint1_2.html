Problem analysis

Per fare ciò decidiamo insieme al committente che è utile, ai fini di un testing più agevole, creare una classe java da usare come MockObj per la serviceaccessgui.java, che:

  • all'avvio mostri all'utente il menù seguente: 
    
            === Menu Cold Storage ===
            1 - Richiedi ticket deposito
            2 - Verifica ultimo ticket
            3 - Verifica ticket specifico
            4 - Mostra ticket attivi
            0 - Esci
            Scelta:
    gestendo la generazione di diversi tipi di test in maniera più semplice ed agevole, rendendo il programma interattivo e funzionale.

    Test plans

    I test da controllare sono i casi:

  • Che i Ticket vengano generati, accettati o rifiutati e mostrati correttamente all'utente.
  • Che i ticket ottenuti siano correttamente validati o eliminati.
  • Che sia possibile richiedere più ticket contempora di seguito, e validarli in tempi differenti.

    • Project

    Queste migliorie ci hanno portato alla architettura in figura:

    Al seguente link sprint.qak è presente il codice qak del progetto migliorato, mentre serviceaccessgui.java è il codice java della classe discussa in precedenza, da far partire insieme al progetto per generare i test.

    Per poter generare i test del progetto è necessario prima far partire basicrobot23 con il comando: docker compose -f basicrobot23.yaml up

    Successivamente far partire sprint.qak da eclipse col comando: ./gradlew run

    Ed infine far partire la classe serviceaccessgui.java con il comando: ./gradlew runserviceaccessgui
